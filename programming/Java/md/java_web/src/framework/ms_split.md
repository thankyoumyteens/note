# 你参与的微服务项目是怎么拆服务的？有没有遇到过拆得过细或过粗的问题？

以 xx 项目为例，整体采用 Spring Cloud 体系，服务拆分主要遵循两个原则：

1. **按业务边界和团队边界拆分**
   - 比如用户、认证、消息中心、钱包等服务，各自有比较清晰的业务边界和团队负责；
   - 边界清晰的业务优先拆分成独立服务，避免出现「大杂烩式」的巨石服务。
2. **按变更频率和复用度拆分**
   - 变更频次高且逻辑相对独立的模块，更适合作为单独服务，减少跨业务方的联动发布；
   - 公共能力（比如统一用户中心、统一字典、消息通知）拆成公共服务或平台类服务，多项目复用。

过程中也遇到过**拆得过细**的问题，比如早期把某些配置查询类的功能单独拆成了一个 service，结果：

- 调用链变长，增加了网络开销和可用性风险；
- 维护成本提升，但业务本身没有多大收益。

后面我们做了一些合并，把这类「业务耦合度很高、变更频次也不算低」的功能适当合并回核心业务服务里，并通过内部模块划分来保持代码结构的清晰。
