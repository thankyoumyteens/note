# 生成 100 年不重复的12位流水号

由于 12 位空间非常有限，我们不能使用标准的 Snowflake 算法（通常需要 19 位）。

## 1. 12 位结构设计

为了兼顾时间顺序、唯一性和并发性，建议结构如下：

| 组成部分           | 长度 | 说明                                                                    |
| ------------------ | ---- | ----------------------------------------------------------------------- |
| **日期偏移量**     | 5 位 | 从基准日期（如 2026-01-01）开始的天数。10^5 天可覆盖 273 年。           |
| **Redis 自增序列** | 6 位 | 每天重置，支持单日 10^6（100万）个流水号。                              |
| **校验位/预留位**  | 1 位 | 用于简单的模运算校验，或作为机房标识（防止极端的 Redis 时钟回拨问题）。 |

**示例流水号：`000630004561**`

- `00063`: 基准日后的第 63 天。
- `000456`: 当日第 456 个流水号。
- `1`: 随机数或校验位。

---

## 2. 核心实现逻辑

### Redis 策略

利用 Redis 的 `INCR` 命令实现原子自增，并结合 `EXPIRE` 设置 24 小时过期，保证每天计数器重置。

### 代码实现思路（Java）

```java
@Service
public class SequenceService {

    @Autowired
    private StringRedisTemplate redisTemplate;

    // 基准日期：2026-01-01
    private static final LocalDate BASE_DATE = LocalDate.of(2026, 1, 1);

    public String generateOrderSn() {
        // 1. 计算天数偏移量 (5位)
        long days = ChronoUnit.DAYS.between(BASE_DATE, LocalDate.now());
        String datePart = String.format("%05d", days);

        // 2. Redis 自增序列 (6位)
        String key = "seq:order:" + datePart;
        Long increment = redisTemplate.opsForValue().increment(key);

        // 第一次生成时设置过期时间（24小时即可）
        if (increment == 1) {
            redisTemplate.expire(key, 25, TimeUnit.HOURS);
        }

        // 如果溢出（超过999999），此处需根据业务抛出异常或扩展逻辑
        String seqPart = String.format("%06d", increment % 1000000);

        // 3. 校验位/随机位 (1位)
        int extraPart = new Random().nextInt(10);

        return datePart + seqPart + extraPart;
    }
}

```

优势：

- **绝对唯一性**：基于天数偏移量，只要 Redis 不停机或自增序列不溢出，100 年内不会重复。
- **趋势递增**：前 5 位是日期，保证了数据库索引的友好性，查询效率高。
- **高性能**：Redis 内存操作，支持高并发下单。

潜在风险与应对：

- **并发瓶颈**：单日 100 万个号。如果业务量极大，可将第 12 位作为“分片位”，从 Redis 的多个 Key 中取号，将上限提升至 1000 万/日。
- **时钟回拨**：如果服务器时间大幅回退，可能产生重复。**对策**：在生成逻辑中判断，如果当前天数小于 Redis 中记录过的最大天数，拒绝服务。

## 什么是“时钟回拨”？

**时钟回拨（Clock Rollback/Jump Back）**指的是服务器的系统时间由于某种原因，突然倒退回了之前的一个时间点。常见的触发原因包括：

- NTP 同步机制：服务器定期与网络时间协议（NTP）服务器同步时间。如果服务器本地时间跑得太快，同步后时间就会被强行拉回过去。
- 闰秒调整：国际自转服务（IERS）偶尔会增加或减少一秒，导致系统时间回拨。
- 虚拟机迁移或宿主机问题：在云环境中，虚拟机迁移或宿主机时钟异常可能导致系统时间突变。

在实际业务中，为了生成可读性高、带有时间趋势且全局唯一的业务 ID（例如订单号），我们通常采用这样的公式：

```
分布式 ID = 业务前缀 + 时间戳 + Redis INCR 生成的自增序列号
(例如：ORDER_20260311120000_000001)
```

为了防止自增序列号无限膨胀，通常会按时间维度（如按天、按秒）给 Redis 的 Key 设置过期时间（TTL）。比如每秒使用一个新的 Key：INCR order_seq:20260311120000。

时钟回拨破坏唯一性的场景如下：

场景 1：应用服务器（App Server）时钟回拨

1. 正常运行：当前时间是 12:00:05，应用向 Redis 发起 INCR seq:12:00:05，返回 1。生成的 ID 是 120005_1。
2. 发生回拨：应用服务器的时钟突然回拨了 5 秒，系统时间变成了 12:00:00。
3. 时间流逝：系统时间再次走到 12:00:05。
4. 发生重复：应用再次向 Redis 发起请求。此时如果之前 seq:12:00:05 的 Key 已经因为设置了 1 秒的过期时间（TTL）而被 Redis 删除了，那么 Redis 会重新创建一个 Key，INCR 再次返回 1。
5. 灾难降临：应用再次生成了 ID 120005_1，唯一性被打破，产生脏数据或主键冲突。

场景 2：Redis 服务器（Redis Server）时钟回拨影响 TTL

Redis 的过期键回收（尤其是旧版本）部分依赖于系统的绝对时间（Wall Clock）。如果在 Redis 中生成按时间段的 Key（如按天），且 Redis 服务器发生时钟回拨：

- 原本应该过期的 Key 可能会因为时间变“早”了而延长存活时间。
- 或者如果时间突进，Key 可能会提前过期被销毁，导致下一个请求到来时序列号又从 1 开始计算，配合上游业务的时间戳，就有可能拼凑出重复的 ID。
