# Java21 以下的版本实现事件循环模型

在 Java 21（引入虚拟线程 Virtual Threads 之前），Java 语言本身并没有像 Python 或 Node.js 那样提供内置的单线程事件循环机制（也没有 `async/await` 语法糖）。

但是，在应对高并发 I/O 场景时，Java 生态早就通过 **NIO (Non-blocking I/O)** 和 **Reactor 模式** 实现了极其成熟且高性能的事件循环架构。

要在 Java 21 以下的版本实现或使用事件循环模型，通常有以下几个层面的实现方式：

### 1. 底层基石：Java NIO (原生 API)

Java 1.4 引入的 `java.nio` 包是 Java 实现事件循环的底层核心。它通过**多路复用（Multiplexing）**技术，允许一个或少数几个线程管理成千上万个网络连接。

它的核心组件包含三个：

- **Channel（通道）：** 类似于传统 I/O 的 Stream，但是非阻塞的。
- **Buffer（缓冲区）：** 数据读写的临时容器。
- **Selector（选择器）：** 这就是 Java 版的事件循环核心。一个线程可以调用 `Selector.select()` 方法，该方法会调用操作系统的底层 API（如 Linux 的 `epoll` 或 Mac 的 `kqueue`）。线程会在这里等待，直到注册在 Selector 上的某个 Channel 发生了事件（如“有数据可读”、“连接就绪”），然后线程被唤醒去处理这些就绪的事件，处理完后继续循环。

_注意：直接编写原生 NIO 代码极其复杂，容易遇到诸如空轮询导致 CPU 100% 的经典 Bug，因此在实际业务开发中极少直接手写 NIO。_

### 2. 工业标准：Netty 框架 (推荐)

为了解决原生 NIO 的复杂性，**Netty** 框架应运而生。它是 Java 生态中实现事件循环和异步网络编程的绝对霸主（几乎所有热门中间件的底层网络通信都在用它）。

Netty 完美地实现了 **Reactor 模式**（事件循环的高级封装），并提供了明确的 `EventLoop` API：

- **BossGroup (主事件循环)：** 通常是一个单线程的事件循环，专门负责监听端口，接收新的客户端连接，并将连接分配给 Worker。
- **WorkerGroup (从事件循环组)：** 包含多个 `EventLoop`（通常等于 CPU 核心数 x 2）。每个 `EventLoop` 都是一个绑定了单个线程的无限循环，负责处理被分配到的连接的所有 I/O 读写事件。

**特点：** 与 Python 严格的单线程事件循环不同，Netty 默认采用的是**多线程事件循环模型**。它既保留了非阻塞 I/O 的高并发能力，又利用了多核 CPU 的计算资源。

### 3. 应用层框架：Spring WebFlux / Project Reactor

如果你是在开发 Web API 层面，传统的 Spring Web MVC 是基于多线程阻塞模型（Servlet 栈）的。要切换到事件循环模型，你需要使用 **Spring WebFlux**。

WebFlux 底层默认就是基于 Netty 运行的，它通过 **Project Reactor** 提供了响应式编程（Reactive Programming）的能力。

- 你不再返回普通的对象，而是返回 `Mono<T>`（代表 0 或 1 个异步结果）或 `Flux<T>`（代表多个异步结果）。
- 当你的代码发起数据库查询或外部 API 调用时（需配合支持响应式的数据库驱动，如 R2DBC），当前线程不会阻塞等待，而是直接返回一个 `Mono`。当底层 I/O 响应到达时，Netty 的事件循环线程会触发回调，继续执行后续的数据转换操作。

### 4. 异步编排：CompletableFuture

在不引入完整的响应式框架时，Java 8 引入的 `CompletableFuture` 是实现异步非阻塞逻辑的利器。虽然它本身不是一个事件循环服务器，但它可以让你在多线程环境中以类似“回调链”的方式编写代码，避免线程被同步阻塞。

你可以将 I/O 任务提交给特定的线程池，并在任务完成时触发 `.thenApply()` 或 `.thenCompose()`，从而模拟出非阻塞的异步处理流。

---

**总结建议：**

在 Java 21 之前，如果你需要构建极高并发的网关、推送服务或游戏服务器，直接使用 **Netty** 是最佳选择。如果你是在开发普通的业务 Web API 并且遭遇了 I/O 瓶颈，可以考虑迁移到 **Spring WebFlux**。

不过，响应式编程（如 WebFlux 的 `Mono/Flux`）的学习曲线非常陡峭，代码调试和异常追踪也比同步代码困难得多。这也是为什么 Java 21 最终推出了**虚拟线程（Virtual Threads）**——它让你能用传统“同步阻塞”的简单代码结构，在底层自动实现非阻塞的事件循环调度，兼顾了开发体验与高并发性能。
