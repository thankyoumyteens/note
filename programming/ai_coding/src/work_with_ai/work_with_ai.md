# 指挥 AI 工作

## 第 1 阶段：从零启动项目

**目标：在没有任何本地项目的情况下，指挥 AI 创建可运行的 Java Spring Boot 项目。**

这一阶段解决的是最基础的问题：

> 我本地什么都没有，怎么让 AI 从空目录创建一个能跑、能测、能继续扩展的 Java 项目？

你不是直接让 AI 写一个“大系统”，而是先建立最小可运行基线。

这个阶段包括：

```text
第 1 课：把“我要一个 Java 项目”变成清晰任务
第 2 课：建立 AGENTS.md / CLAUDE.md
第 3 课：初始化 Git 与基线提交
```

它训练三种能力：

第一，**控制 AI 不过度设计**。

从零项目最容易被 AI 直接加上数据库、Security、Docker、Redis、Swagger、复杂分层。第 1 阶段要求你把范围压到最小：

```text
Spring Boot
Maven
健康检查接口
基础测试
README
```

第二，**建立项目规则文件**。

`AGENTS.md` 和 `CLAUDE.md` 的作用是告诉 Codex / Claude Code：

```text
技术栈是什么
测试命令是什么
不要乱加依赖
不要硬编码密钥
非平凡修改先计划
行为变更要补测试
```

这相当于给 AI 建立项目纪律。

第三，**建立 Git baseline**。

这是风险控制点。后面 AI 一旦乱改，你能用：

```bash
git diff
git status
git restore
git reset
```

判断和回滚。

这一阶段完成后，你拥有的是：

```text
一个干净、可运行、可测试、可回滚的项目起点
```

这是后续所有课程的地基。

## 第 2 阶段：基础指令能力

**目标：学会把模糊需求变成 AI 可执行任务，并让 AI 输出可审查结果。**

这一阶段开始进入业务需求，但还不是马上实现。

它解决的问题是：

> 我怎么把“帮我做个功能”这种模糊话，变成 AI 能稳定执行、不乱发挥的任务？

这个阶段包括：

```text
第 4 课：从模糊需求到清晰任务
第 5 课：输出格式控制
```

核心训练是 **6 段式任务表达**：

```text
目标：
背景：
输入：
输出：
限制：
验收标准：
```

比如模糊需求是：

```text
帮我做一个文档保存功能。
```

清晰任务则要变成：

```text
目标：新增 POST /api/documents
背景：当前项目已有最小 Spring Boot baseline
输入：当前代码、规则文件、README
输出：先输出计划，不改代码
限制：不接数据库、不接 AI、不加用户系统、不加 Security
验收标准：title/content 非空，成功返回 documentId，有测试计划
```

第 5 课进一步训练“输出格式控制”。

同一个需求，你要让 AI 输出不同材料：

```text
产品解释
开发计划
接口 JSON
测试清单
Review checklist
```

这很重要。真实工作里，你不会只让 AI “写代码”。你还会让它帮你：

```text
解释需求
设计接口
列测试点
准备 review checklist
整理交付说明
```

这一阶段完成后，你能做到：

```text
把模糊需求变成清晰任务
让 AI 按指定格式输出可审查材料
避免 AI 输出一大段不可执行的散文
```

## 第 3 阶段：Plan-Then-Act 与小步实现

**目标：先让 AI 制定方案，再决定是否执行；实现时保持小 diff。**

这是从“会提需求”进入“会管执行”的阶段。

它解决的问题是：

> AI 写代码很快，但怎么防止它一次性乱改一堆文件？

这个阶段包括：

```text
第 6 课：先计划，后执行
第 7 课：小步实现文档保存功能
```

第 6 课只做 Plan，不做 Act。

你让 AI 输出：

```text
需求理解
涉及文件
修改步骤
数据结构 / API 变化
测试计划
风险和回滚方式
计划自检
```

这一步的价值是：**在代码被修改前审查 AI 的意图**。

你要检查：

```text
它准备改哪些文件？
有没有想改 pom.xml？
有没有想新增依赖？
有没有想加数据库？
有没有忘记成功响应只返回 documentId？
有没有测试计划？
有没有回滚方式？
```

第 7 课才进入小步实现。

小步实现的原则是：

```text
一次只做一个任务
每步说明改了哪些文件
每步后运行测试
不修改无关文件
遇到不确定点先停下来
```

比如文档保存功能可以拆成：

```text
1. 新增 DTO
2. 新增 Document 模型
3. 新增 InMemoryDocumentStore
4. 新增 Controller
5. 新增 MockMvc 测试
6. 运行 mvn test
```

这一阶段完成后，你掌握的是：

```text
不让 AI 直接冲进代码
先审计划
再小步实现
每一步都可 review、可测试、可回滚
```

这是 AI coding agent 最关键的工程习惯之一。

## 第 4 阶段：Spec Workflow

**目标：用轻量规格文件管理中大型功能。**

这一阶段解决的是：

> 当功能变复杂后，单条 prompt 不够了，怎么让 AI 长期围绕同一套需求工作？

答案是轻量 Spec。

它包括：

```text
第 8 课：轻量 Spec 结构
第 9 课：Spec 驱动实现文档查询功能
```

Spec 不是传统那种很重的瀑布文档，而是给 AI 的稳定上下文。

典型结构是：

```text
specs/document-query/
  requirements.md
  design.md
  tasks.md
  test.md
```

每个文件分工不同：

```text
requirements.md：要做什么、不做什么、验收标准
design.md：怎么做、影响哪些文件、风险是什么
tasks.md：一步步执行任务
test.md：怎么验证
```

这个阶段实战功能是：

```text
GET /api/documents/{id}
```

也就是文档查询功能。

你会训练 AI：

```text
先读 spec
检查 spec 是否矛盾
按 tasks.md 逐项实现
每完成一项更新任务状态
不要实现 Non-goals
最后更新 test.md
```

Spec Workflow 的价值是：

```text
减少上下文丢失
减少 AI 自由发挥
让复杂功能可以拆解和追踪
让需求、设计、任务、测试保持同步
```

这一阶段完成后，你会从“靠一次 prompt 管 AI”升级为：

```text
用规格文件管理 AI 的长期执行
```

## 第 5 阶段：TDD with AI

**目标：用测试驱动 AI，减少“看似正确”的代码。**

这一阶段解决的是：

> AI 写出来的代码看起来对，但怎么证明它真的对？

答案是先写测试。

它包括：

```text
第 10 课：先写测试，不写实现
第 11 课：最小实现让测试通过
```

实战功能是：

```text
POST /api/documents/{id}/summary
```

也就是文档摘要生成接口。

但注意：这一阶段仍然不接真实 AI API，而是先用：

```text
FakeAiSummaryClient
```

第 10 课要求 AI 只写测试，不写实现。

测试覆盖：

```text
文档存在时生成摘要成功
文档不存在时返回 404 或明确错误
重复生成时行为明确
AI client 抛异常时有明确错误处理
```

第 11 课再让 AI 做最小实现，让测试通过。

这里有两个关键训练点。

第一，**先定义正确行为，再写代码**。

否则 AI 可能写一个“能跑但行为不清楚”的实现。

第二，**Fake client 先行**。

真实大模型 API 不稳定、慢、有成本，还涉及密钥。课程先用 fake 实现，保证业务流程和测试可控。

这一阶段完成后，你掌握的是：

```text
先让 AI 写测试
用失败测试约束实现
只做让测试通过的最小代码
避免 AI 写出大段无验证代码
```

这是把 AI 编程从“生成代码”拉回“工程验证”的关键。

## 第 6 阶段：外部 AI API 接入

**目标：在已有 Fake client 的基础上，安全接入真实大模型 API。**

这一阶段才正式接入真实模型。

它解决的是：

> 怎么把 OpenAI / Claude / 其他模型接入 Java 后端，同时不污染业务逻辑、不泄露密钥、不让测试依赖真实网络？

它包括：

```text
第 12 课：真实 AI Provider 设计
第 13 课：实现真实 AI Provider
```

推荐结构是：

```text
AiSummaryClient        # 接口
FakeAiSummaryClient    # 本地测试实现
OpenAiSummaryClient    # 真实 provider
AiProperties           # 配置类
SummaryService         # 业务服务，只依赖接口
```

这里最重要的原则是：

```text
Controller 不直接调用外部 API
Service 不绑定具体供应商
测试默认使用 Fake provider
API key 从配置读取
不要把密钥写进代码、测试、README
真实 provider 只能配置启用
```

这个阶段会训练你处理真实 AI 应用开发里的核心工程问题：

```text
Provider 抽象
配置管理
密钥安全
超时
错误处理
测试隔离
供应商替换
```

这一阶段完成后，你的项目会从“模拟 AI 功能”升级为：

```text
具备可控、安全、可测试的真实 AI Provider 接入能力
```

## 第 7 阶段：AI Code Review

**目标：让 AI 帮你审查代码，但最终由人判断。**

这一阶段解决的是：

> AI 可以写代码，那它能不能也帮我审代码？可以，但人必须最终判断。

它包括：

```text
第 14 课：功能审查与回归审查
第 15 课：Java 后端专项审查
```

第 14 课是通用 review：

```text
是否满足需求
是否有隐藏 bug
是否破坏已有行为
是否缺少测试
是否有安全风险
是否过度设计
是否有更小实现方案
```

第 15 课是 Java 后端专项 review：

```text
Controller 是否过重
Service 边界是否清晰
DTO/API 是否破坏兼容性
异常处理是否合理
日志是否泄露敏感信息
外部 API 超时和错误处理是否安全
配置中是否可能泄露密钥
测试是否覆盖关键路径
```

这个阶段非常重要，因为 AI 经常会写出：

```text
表面能跑
但边界条件不足
异常处理粗糙
测试覆盖不完整
响应结构不稳定
未来扩展困难
```

你要学会让 AI 作为 reviewer，而不是只作为 coder。

但课程也明确：**最终由人判断**。

AI 的 review 是辅助，不是替你负责。

这一阶段完成后，你能做到：

```text
用 AI 审查 AI 的代码
用 checklist 降低遗漏
把 review 重点对准 Java 后端真实风险
```

## 第 8 阶段：重构、文档与长期维护

**目标：把项目从“能跑”升级为“可维护”。**

最后一阶段解决的是：

> 项目功能能跑之后，怎么变得可维护、可交接、可持续迭代？

它包括：

```text
第 16 课：安全小步重构
第 17 课：生成项目文档和 Handoff
第 18 课：Commands / Skills / Hooks 自动化
```

第 16 课训练安全重构。

重点不是让 AI 大改架构，而是先调查：

```text
哪里重复？
哪里 Controller 太重？
哪里 Service 边界不清？
哪里可以小步优化？
风险是什么？
最小重构步骤是什么？
```

核心原则是：

```text
不要为了设计模式而设计模式
不要扩大业务范围
重构后测试必须通过
```

第 17 课训练文档和 handoff。

你会让 AI 生成：

```text
README 更新
API 文档
架构说明
HANDOFF.md
```

`HANDOFF.md` 的价值很高，尤其是长会话或换工具时。它告诉新会话：

```text
已完成什么
修改了哪些文件
还剩什么
当前阻塞点
关键设计决策
如何继续
应该运行哪些测试
```

第 18 课训练自动化。

比如：

```text
修改后运行测试
提交前生成 review checklist
根据 spec 执行下一项任务
生成 handoff
更新 COURSE.md 学习记录
```

如果用 Claude Code，后续可能会映射到：

```text
commands
skills
hooks
```

如果用 Codex，也可以沉淀成项目工作流文档或脚本。

这一阶段完成后，你的项目不只是“写完了”，而是具备：

```text
可维护性
可交接性
可审查性
可持续迭代能力
```

## 这 8 个阶段的整体逻辑

可以把它们理解成一条能力升级线：

```text
第 1 阶段：先有一个干净项目
第 2 阶段：学会把需求说清楚
第 3 阶段：学会控制 AI 执行过程
第 4 阶段：学会用 spec 管复杂功能
第 5 阶段：学会用测试约束 AI
第 6 阶段：学会接入真实 AI 能力
第 7 阶段：学会让 AI 帮你审查质量
第 8 阶段：学会长期维护和交接
```

更直白地说：

```text
能启动 → 能表达 → 能计划 → 能实现 → 能验证 → 能接真实模型 → 能审查 → 能维护
```
