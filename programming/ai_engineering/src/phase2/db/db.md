# 向量数据库落地

如果说前三步我们在搞“算法和数据处理”，那么这一步我们就正式进入了 **“工程化落地”**。把数据存在内存里（像之前的 Chroma 默认模式）是不能上生产的，一旦宕机数据全丢，而且也无法应对高并发。

我们需要一个真正的向量数据库 (Vector Database)。

## 传统关系型数据库 vs 向量数据库

| 传统关系型数据库 (MySQL)     | 向量数据库 (Qdrant / Milvus)                  | 对应的 RAG 概念                                                            |
| :--------------------------- | :-------------------------------------------- | :------------------------------------------------------------------------- |
| **Database / Table** (表)    | **Collection** (集合)                         | 存放某一类文档的容器（如“员工手册集合”）                                   |
| **Column** (列)              | **Dimension** (维度) / **Payload** (标量字段) | 维度是向量的长度（如 BGE 是 512 维）；Payload 是 Metadata（如“章节=考勤”） |
| **Row** (行)                 | **Point / Entity** (实体)                     | 一个文本块（Chunk）+ 它的向量 + 它的 Metadata                              |
| **B+ Tree Index** (B+树索引) | **HNSW Index** (分层导航小世界索引)           | 加速查找的底层数据结构                                                     |
| `ORDER BY` (排序)            | **Distance Metric** (距离度量)                | 核心算法（通常是**余弦相似度 Cosine**），算出谁离问题最近                  |
