# Token 与 Tokenizer

LLM 不直接按字数或单词数处理文本，而是先通过 Tokenizer 将文本转换成 Token，再转换成模型能够计算的 Token ID。

## Token

Token 是模型处理文本的基本单位，但不固定等于一个字、一个单词或一个字符。它可能是完整单词、子词、汉字、标点、空格与文本片段的组合。

同一段文本使用不同模型的 Tokenizer 时，切分结果和 Token 数量可能不同。中文、英文、代码、数字和特殊符号的切分规律也不相同，因此不能用字符数或单词数精确推断 Token 数量。

## Tokenizer

Tokenizer 负责完成两个方向的转换：

1. 编码：将输入文本切分成 Token，并映射为词表中的 Token ID。
2. 解码：将模型生成的 Token ID 还原为可读文本。

模型实际接收和输出的是 Token ID。Tokenizer 的词表与切分规则属于模型契约的一部分，必须与模型匹配；使用错误的 Tokenizer 估算长度，结果可能与 Provider 的实际统计不一致。

## 常见切分方法

### BPE

BPE 从较小的文本单元开始，反复合并高频组合形成子词。常见词可以使用较少 Token 表达，少见词仍能拆成更小单元，兼顾词表大小和未知词处理。

### WordPiece

WordPiece 同样使用子词词表，但合并时更关注哪种组合能提升语言数据的建模效果。它常见于 BERT 系列模型，通常会用特殊形式标记某个子词是否接在词语中间。

### SentencePiece

SentencePiece 直接基于原始文本训练，不要求先按空格分词，能够统一处理中文、英文等不同语言。它是 Tokenizer 工具和建模方式，可以在内部使用 BPE 或 Unigram 等算法。

这些方法的共同目标不是还原语言学意义上的“词”，而是在有限词表下，把任意文本稳定转换成模型可处理的 ID 序列。

## 对应用开发的影响

- 模型、Provider 或 Tokenizer 发生变化时，同一请求的 Token 数量可能变化。
- 代码、JSON、长数字和混合语言文本可能产生与直觉不同的 Token 数量。
- 本地 Tokenizer 适合请求前估算，Provider 响应中的 usage 才是计费和调用记录的最终依据。
- Token 数量会进一步影响上下文窗口、截断、延迟和成本，具体见 [Token 限制及其工程影响](./token_limit.md)。
