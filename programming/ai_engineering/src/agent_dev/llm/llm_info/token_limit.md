# Token 限制及其工程影响

一次模型调用通常包含输入 Token 和输出 Token。输入包括 System Prompt、历史消息、当前用户请求、RAG 检索内容和工具结果；输出是模型本次生成的内容。

## 上下文窗口

上下文窗口是模型一次调用能够处理的 Token 总量。通常可以理解为：

> 输入 Token + 可生成的输出 Token 不能超过模型支持的上下文窗口。

请求参数中的 `max_tokens`、`max_completion_tokens` 或 `max_output_tokens` 用于限制最大输出长度，不会扩大上下文窗口。为输出预留的空间不足时，即使输入本身没有超过窗口，也可能无法生成完整结果。

## 超出限制

输入或输入与预期输出的总量超过限制时，Provider 可能直接拒绝请求，也可能由应用或框架预先截断内容。截断如果没有明确优先级，可能删除 System Prompt、关键历史消息、检索证据或用户当前请求，导致回答偏离约束。

应用不应依赖隐式截断，而应在调用前明确处理：

- 限制单次用户输入和工具结果长度。
- 为模型输出预留 Token 空间。
- 按优先级裁剪历史消息和检索内容。
- 对长对话进行摘要，但保留关键事实和当前任务状态。
- 超出安全范围时返回明确错误，而不是静默丢弃关键内容。

阶段一只需建立长度边界并记录 usage；复杂的上下文选择、排序、摘要和裁剪属于后续 Context Engineering。

## 延迟

输入越长，模型读取和处理上下文所需的计算通常越多，首 Token 延迟可能上升。输出越长，需要执行的逐 Token 生成步骤越多，完整响应时间通常越长。

延迟不能只看总耗时。流式调用还需要分别关注首 Token 延迟和后续生成速度；非流式调用则主要记录完整响应耗时。

## 成本

Provider 通常分别统计输入 Token 和输出 Token，并可能使用不同单价。完整成本还可能受到模型、缓存命中、推理 Token 或批处理方式影响，应以 Provider 当前计费规则和响应 usage 为准。

工程上需要记录输入、输出和总 Token usage，并将它们与 provider、模型、请求结果和响应延迟关联。字符数只能用于粗略保护，不能替代模型对应 Tokenizer 的调用前估算，也不能替代 Provider 返回的最终 usage。
