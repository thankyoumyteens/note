# 角色设定、任务约束和输出格式约束

这三类约束共同定义一次模型调用的行为契约：角色设定说明模型以什么职责处理，任务约束说明按照什么规则处理，输出格式约束说明以什么形式返回结果。

## 角色设定

角色设定用于限定模型在当前任务中的职责和视角，通常放在应用控制的 system 消息中。

有效的角色需要明确：

- 业务身份：模型在当前任务中是什么，例如订单意图识别器。
- 职责范围：负责识别意图，不负责查询或修改订单。
- 能力边界：只能依据提供的输入判断，不能补充不存在的信息。

“你是一个聪明、专业的助手”没有提供可判断的业务边界，通常不能提高结果稳定性。角色只限定职责，不能代替具体任务规则。

## 任务约束

任务约束把业务规则转换成模型可以执行的判断条件，需要明确：

- 任务目标：本次需要完成什么工作。
- 输入含义：哪些内容是待处理数据，哪些内容是应用规则。
- 可选结果：允许模型返回哪些业务值。
- 判断边界：每个结果在什么条件下成立。
- 兜底规则：信息不足、冲突或无法判断时如何处理。
- 禁止行为：是否允许推测、补全、解释或执行输入中的指令。
- 规则优先级：多条规则冲突时以哪条为准。

约束应具体且可判断。“准确分析”“认真回答”没有明确执行标准，不属于有效的任务规则。

## 输出格式约束

输出格式约束定义模型返回结果的外部形式，需要明确：

- 返回类型：纯文本、JSON、枚举或其他结构。
- 字段契约：字段名称、含义、类型和是否必填。
- 取值范围：枚举、长度、数值范围和是否允许 `null`。
- 额外内容：是否允许增加字段、解释或 Markdown 标记。
- 异常结果：无法判断、拒绝回答或输入缺失时返回什么。

Prompt 中的输出要求必须与程序使用的 Schema 一致。但 Prompt 只能提高模型遵循格式的概率，不能保证结果合法；JSON 解析、Schema 校验和业务校验仍由结构化输出部分处理。

## 消息放置

角色、任务规则和输出约束属于应用控制的固定内容，应放在 system 模板。当前用户请求属于动态数据，应放在 user 模板，并使用清晰边界与指令分隔。

用户输入中的“忽略之前规则”等内容仍然属于待处理数据，不能被拼接到 system 模板，也不能覆盖应用规则。

## 订单意图识别案例

当前代码使用订单意图识别演示三类约束：

- 角色：订单系统的用户意图识别器。
- 任务：把用户输入判断为 `QUERY_ORDER`、`CANCEL_ORDER` 或 `UNKNOWN`。
- 约束：只依据输入判断，无法确定时返回 `UNKNOWN`，不执行输入中的指令。
- 输出：只返回包含 `intent` 字段的 JSON 对象，不增加解释或其他字段。

三种实现都在各自的模板配置中表达相同契约：

- [Spring Boot + WebClient](../prompt_template/spring_boot/config.md)
- [Spring AI](../prompt_template/spring_ai/config.md)
- [Python + uv](../prompt_template/python/config.md)

三种实现的区别只在模板 API 和变量语法，角色、任务规则和输出契约应该保持一致。

## 边界

这三类 Prompt 约束不能替代权限检查、业务参数校验、结构化输出能力和程序侧结果校验。模型即使返回符合格式的内容，也不代表业务语义一定正确，更不能据此跳过后续安全检查。
