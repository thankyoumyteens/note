# Prompt

Prompt 是应用发送给模型的指令和上下文，用来说明模型需要完成什么任务、可以使用哪些信息、必须遵守哪些规则以及应该返回什么结果。

在工程中，Prompt 不是临时拼接的一段自然语言，而是模型调用契约的一部分。清晰的 Prompt 可以减少任务理解偏差，但不能保证模型一定生成合法、正确的结果，输出仍然需要结构化约束和程序校验。

## Prompt 的组成

一个面向业务任务的 Prompt 通常包含：

- 角色：模型在当前任务中承担的职责。
- 任务：需要模型完成的具体工作。
- 输入：用户请求、业务数据和必要上下文。
- 约束：允许执行的范围、禁止行为和判断规则。
- 输出要求：返回内容、格式、字段含义和无法判断时的处理方式。
- 示例：通过输入与输出示例说明抽象规则。

并非所有任务都需要完整包含这些部分。只保留会影响模型判断和输出的内容，避免加入重复背景、无关规则和无法验证的描述。

## 消息职责

Prompt 通常通过 system 和 user 消息发送：

- system 消息保存由应用控制的角色、任务规则和输出约束。
- user 消息保存本次请求和与请求直接相关的数据。
- assistant 消息用于多轮对话历史或 Few-shot 示例中的预期输出。

应用规则和用户输入需要明确分离。用户提供的内容应被当作待处理数据，不能直接拼接成 system 规则，避免用户输入覆盖应用约束。消息角色的基础含义见 [system / user / assistant 消息结构](../../llm_api/message.md)。

## 约束设计

约束应具体、可判断，重点说明：

- 任务范围和判断条件。
- 可选值、边界条件和优先级。
- 信息不足或无法判断时返回什么。
- 是否允许解释、推测或补全缺失信息。
- 输出必须包含哪些内容。

“准确回答”“认真分析”这类要求没有明确判定标准，通常不能显著提高可靠性。存在冲突的规则应明确优先级，否则模型可能在不同调用中采用不同解释。

## Few-shot 示例

Few-shot 是在 Prompt 中提供少量输入与预期输出，帮助模型理解分类边界、字段含义和特殊情况。它适合难以仅用规则描述，或者不同类别容易混淆的任务。

示例应覆盖有代表性的正常情况和关键边界，格式必须与真实输入输出一致。示例过多会增加 token、延迟和维护成本，也可能让模型过度模仿示例中的非关键内容。

## Prompt 模板

固定规则与动态输入应通过模板分离，使 Prompt 可以复用、审查和测试。模板负责定义稳定结构，渲染过程负责填充业务变量并生成最终消息。

动态值需要按模板规则安全填充，并使用清晰的边界标识区分指令与数据。不能依赖字符串替换处理复杂嵌套结构，也不能把未经处理的用户输入解释为新的应用指令。

## 版本与变更

Prompt 应具有稳定名称和版本。只要任务规则、输出要求、示例或变量语义发生变化，就需要记录变更，并使用固定测试集比较变更前后的结果。

Prompt 版本应与模型、模型参数和输出 Schema 一起记录。相同 Prompt 在不同模型或参数下可能产生不同结果，只记录 Prompt 文本无法完整复现一次调用。

## Prompt 的边界

Prompt 只能提高模型遵循要求的概率，不能替代：

- JSON Schema 等模型侧结构化输出能力。
- 程序侧的解析、Schema 校验和业务校验。
- 权限检查、参数校验和安全策略。
- 对模型拒绝、截断、格式错误和语义错误的处理。

对于程序必须稳定消费的结果，应由 Prompt 说明任务语义，由结构化输出定义返回契约，再由应用代码验证最终结果。
