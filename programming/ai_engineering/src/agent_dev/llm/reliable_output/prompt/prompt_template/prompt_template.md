# Prompt 模板设计

Prompt 模板把稳定的任务规则与每次调用变化的业务数据分开管理，使同一任务可以复用、追踪版本和重复测试。

可以把它理解为：Prompt 模板 = 固定指令 + 输入变量 + 输出要求 + 可选示例。模板只负责组织模型输入，不能替代结构化输出约束和程序校验。

## 固定内容与动态内容

固定内容由应用维护，通常包括角色、任务、判断规则、输出要求和 Few-shot 示例。动态内容来自当前请求，通常包括用户输入、业务数据和检索上下文。

两者需要明确分离：

- 固定规则放在 system 模板中，不能由用户输入修改。
- 当前请求放在 user 模板中，并通过变量填充。
- Few-shot 的示例输入和输出保持成对出现，格式与真实调用一致。

不要把完整 Prompt 作为一个字符串交给业务代码自由拼接，否则规则、数据和消息角色会混在一起，难以审查和复现。

## 模板组成

一个模板通常需要定义：

- 名称：标识模板对应的业务任务，例如意图识别。
- 版本：标识一组确定的规则、变量和示例。
- system 模板：保存角色、任务、约束和输出要求。
- user 模板：描述当前输入，并声明动态变量的位置。
- 变量定义：记录变量名称、含义、是否必填和允许的值。
- 示例：补充容易混淆的分类边界和异常输入。

名称和版本属于调用记录的一部分；仅保存渲染后的文本，无法准确判断结果来自哪个模板定义。

## 变量设计

变量名应表达业务含义，例如 `user_input`、`order_context`，不要使用 `text1`、`data` 这类模糊名称。每个变量只承担一种职责，避免把多段来源不同的数据提前拼成一个变量。

渲染前需要检查必填变量、类型和长度。渲染时应保留明确的数据边界，使模型能够区分应用指令与用户数据；变量内容不能被当作新的 system 规则。

可选变量缺失时，应由模板定义省略对应片段或填入明确的缺省语义，不能把空值、`null` 或未替换的占位符直接发送给模型。

## 输出要求

模板需要说明模型应返回什么、字段代表什么，以及信息不足时如何处理。输出要求应与实际 Schema 保持一致，避免 Prompt 使用一组字段，程序却按另一组字段解析。

只在 Prompt 中展示 JSON 示例不能保证返回合法 JSON。需要程序稳定消费时，还应使用 Provider 支持的结构化输出能力，并在返回后执行 Schema 校验和业务校验。

## Few-shot 示例

示例用于补充规则难以表达的边界，而不是重复正常规则。优先选择容易混淆、信息不足和需要返回兜底值的输入。

示例属于模板版本的一部分。修改示例可能改变模型的分类倾向，因此需要和规则变更一样执行固定测试集回归。

## 渲染流程

一次模板调用可以拆成四步：

1. 根据名称和版本取得模板。
2. 校验并填充动态变量。
3. 分别生成 system、user 和必要的示例消息。
4. 将渲染后的消息交给已有 ProviderClient 调用。

模板渲染只生成消息，不负责选择 Provider、执行重试或解析业务结果。这样可以继续复用“最小 LLM 应用”中的统一调用链。

## 版本与优化

任务规则、变量语义、输出要求或示例发生变化时，应创建新版本，不要直接覆盖正在使用的模板。调用记录至少关联模板名称、版本、模型和关键模型参数。

比较两个模板版本时，应使用同一测试集和同一模型配置，观察任务正确率、格式成功率、关键字段准确率、Token 用量和延迟。A/B 测试用于比较候选版本，不能代替离线回归和结果校验。
