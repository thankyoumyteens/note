# Few-shot 示例设计

Few-shot 是在当前输入前提供少量“示例输入—预期输出”，用具体案例补充难以完全写成规则的判断边界。它适合容易混淆的分类、字段语义和兜底场景，不是每个 Prompt 都必须使用。

## 示例选择

示例应来自真实任务中的稳定模式，优先覆盖：

- 容易混淆的类别边界。
- 信息不足、语义冲突和无法判断的输入。
- 业务上重要但仅靠文字规则不直观的情况。

正常案例只保留理解任务所需的最少数量。不要堆叠含义重复的示例，也不要使用与生产输入格式不同的演示文本。

当前订单意图识别案例使用三组示例：查询订单、取消订单，以及同时表达查询和取消而必须返回 `UNKNOWN` 的冲突输入。第三组示例明确了兜底边界，比重复增加普通查询案例更有价值。

## 消息结构

每组示例使用连续的 `user` 和 `assistant` 消息，最后再追加当前 `user` 消息：

```text
system：角色、任务规则和输出要求
user：示例输入 1
assistant：预期输出 1
user：示例输入 2
assistant：预期输出 2
user：用户真正的输入
```

示例输入应使用与当前输入相同的数据边界，示例输出应与真实输出契约完全一致。示例不是 system 规则，也不能混入当前用户数据。

## 数量与顺序

示例数量由任务边界决定，不存在固定最佳值。先加入能够消除主要歧义的最小集合，再根据固定测试集结果决定是否增加。示例越多，输入 Token、延迟和成本越高，也更容易让模型模仿无关措辞。

顺序可能影响结果。固定示例内容和顺序，并把它们纳入 Prompt 版本；比较示例变更时保持模型、参数和测试集一致。

## 版本与评估

新增、删除、修改或调整示例顺序都属于 Prompt 变更，需要创建新版本。调用记录继续保存 `promptName`、`promptVersion`、模型和参数。

评估时比较加入示例前后的任务正确率、格式成功率、Token 用量和延迟，并重点检查示例未覆盖的输入，避免测试集只验证模型是否记住示例。

三种实现都复用 Prompt 模板章节的统一 DTO、ProviderClient 和 Router，只把示例消息加入已有请求：
