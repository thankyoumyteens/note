# Prompt 版本管理

Prompt 版本管理把一次模型调用使用的固定规则、变量语义、输出要求和 Few-shot 示例保存为不可变版本，使调用结果可以复现、比较和回滚。

## 版本对象

一个完整版本至少包含：

- 稳定的 Prompt 名称，例如 `intent-classifier`。
- 不可变的版本号，例如 `v1`、`v2`。
- system 和 user 模板。
- 与该模板配套的 Few-shot 示例及顺序。

不能只给模板文本编号而把示例放在独立的可变配置中，否则相同版本号可能产生不同消息。模型、模型参数和输出 Schema 不属于 Prompt 文本，但必须与 `promptName`、`promptVersion` 一起进入调用记录。

## 版本规则

已发布版本不能原地覆盖。角色、规则、变量含义、输出要求、示例内容或示例顺序发生变化时，创建新版本；仅修改注释和文档措辞且不影响实际消息时不需要升级。

版本号只表示同一 Prompt 的变更顺序，不表达模型版本，也不在不同 Prompt 名称之间比较大小。应用通过明确的名称和版本查找定义，找不到时直接启动失败或终止调用，不能静默退回其他版本。

## 发布与回滚

当前启用版本由应用配置决定，不由 HTTP 请求参数或用户输入决定。发布新版本时先把新旧版本同时注册，再修改启用版本；回滚只把配置切回仍然保留的旧版本。

本节使用进程内 Registry 演示最小闭环。它适合当前单应用学习案例；多实例动态发布时可以把版本定义放入配置中心或数据库，但仍需在应用内完成完整性校验和缓存一致性处理。

## 调用流程

```text
应用配置中的名称与版本
  -> PromptRegistry 精确查找
  -> 渲染模板
  -> 追加该版本的 Few-shot 示例
  -> 追加当前输入
  -> 调用 Provider Router
  -> 记录实际 promptName 和 promptVersion
```

Registry 只负责保存和精确查找版本，不负责流量分配、自动回滚或评估。Prompt A/B 测试和持续优化在后续章节实现。
