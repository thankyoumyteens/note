# system / user / assistant 消息结构

`system / user / assistant` 是 LLM 对话 API 里的 **消息角色结构**。它的作用是告诉模型：

- 这段话是谁说的？
- 这段话应该被当成规则、用户输入，还是历史回复？
- 模型应该优先听谁的？

在 OpenAI-compatible Chat Completions 里，常见结构是：

```json
{
  "messages": [
    {
      "role": "system",
      "content": "你是一个严谨的 Java 后端助手。"
    },
    {
      "role": "user",
      "content": "解释一下 RAG。"
    }
  ]
}
```

## 整体结构

你可以把一次模型调用理解成：

```text
                    ┌────────────────┐
                    │  system 消息    │
                    │  规则 / 角色    │
                    └────────┬───────┘
                             │
                             ▼
                    ┌────────────────┐
                    │   user 消息     │
                    │   用户输入      │
                    └────────┬───────┘
                             │
                             ▼
                    ┌────────────────┐
                    │ assistant 消息  │
                    │  模型历史回复   │
                    └────────┬───────┘
                             │
                             ▼
                    ┌────────────────┐
                    │      LLM        │
                    │  基于上下文生成  │
                    └────────────────┘
```

更准确地说，`messages` 是一个**有顺序的列表**：

```text
messages[0] → system
messages[1] → user
messages[2] → assistant
messages[3] → user
messages[4] → assistant
以此类推...
```

模型会根据这个列表里的所有内容，判断当前应该怎么回答。

## system 消息

`system` 消息用来定义模型的**角色、规则、行为边界、输出风格**。

例如：

```json
{
  "role": "system",
  "content": "你是一个严谨、清晰的 Java 后端和 AI Agent 开发助手。回答要结构清楚，代码要加简洁注释。"
}
```

它告诉模型：你应该扮演什么角色，以及回答时遵守什么规则。

常见用途：

1. 设定角色
2. 设定回答风格
3. 设定输出格式
4. 设定业务规则
5. 设定安全边界
6. 限制模型不要做什么

例如：

```text
你是一个 Java 后端助手。
回答使用中文。
代码使用 Java 21。
不要引入不必要依赖。
不确定时说明不确定。
```

在工程里，`system` 通常由**应用开发者写死或配置化管理**，不是用户直接传入。

## user 消息

`user` 消息表示**用户输入**。

例如：

```json
{
  "role": "user",
  "content": "帮我解释一下 Spring Boot 中 Service 层的作用。"
}
```

它告诉模型：用户这次真正想问什么、想做什么。

## assistant 消息

`assistant` 消息表示**模型之前的回复**。

例如第一轮对话：

```json
[
  {
    "role": "user",
    "content": "什么是 RAG？"
  },
  {
    "role": "assistant",
    "content": "RAG 是检索增强生成，用外部知识库增强模型回答。"
  }
]
```

第二轮用户继续问：

```json
[
  {
    "role": "user",
    "content": "什么是 RAG？"
  },
  {
    "role": "assistant",
    "content": "RAG 是检索增强生成，用外部知识库增强模型回答。"
  },
  {
    "role": "user",
    "content": "那它和微调有什么区别？"
  }
]
```

这里把上一轮的 `assistant` 回复放进去，是为了让模型知道：

1. 之前已经聊过什么
2. 上一次它自己回答了什么
3. 这一次用户的“它”指的是什么

多轮对话可以由多条不同角色的消息组成，模型生成的消息是 `assistant` 角色。

## 三种角色的对比

| role      | 谁提供        | 主要作用                 | 是否应该让用户直接控制 |
| --------- | ------------- | ------------------------ | ---------------------- |
| system    | 开发者 / 系统 | 设定模型规则、角色、边界 | 不应该                 |
| user      | 真实用户      | 表达用户问题或任务       | 可以                   |
| assistant | 模型          | 保存模型历史回复         | 不应该随便伪造         |
