# 模型与 Provider 的差异和选型

模型选型不是简单比较 GPT、Claude、Llama、Qwen 或 DeepSeek 哪个更强。一个应用最终使用什么能力，同时取决于模型、Provider 和 API 契约。

## 模型、Provider 和 API 契约

### 模型

模型负责理解输入并生成结果。模型之间主要比较：

- 任务能力
- 上下文窗口
- Structured Outputs
- Tool Calling
- 多模态能力
- 延迟
- 成本
- 是否支持私有化部署

同一个 Provider 可以提供多个模型，同一个模型也可能通过不同 Provider 或部署平台提供。

### Provider

Provider 负责将模型作为服务提供出来，包括：

- API 访问
- 鉴权和密钥管理
- 模型版本
- 配额和限流
- 计费
- 服务区域和网络
- 数据处理策略
- SDK 和技术支持

选中了模型，不代表 Provider 也随之确定。例如，开权重模型可以使用托管 Provider，也可以部署在自己的基础设施中。

### API 契约

API 契约决定应用如何调用 Provider，包括 endpoint、鉴权 Header、请求字段、响应结构和错误格式。

模型能力、Provider 服务和 API 契约需要分开判断。模型能力适合某个任务，不代表某个 Provider 的服务质量、价格或接口一定适合；两个 Provider 使用相似接口，也不代表背后的模型能力和运行行为相同。

三类 API 的具体差异见[API 类型](../llm_api/llm_api_type.md)。

## 主流模型家族的基本定位

| 模型家族 | 基本定位 |
| --- | --- |
| GPT | 闭源通用模型，覆盖推理、代码、多模态和工具调用等场景 |
| Claude | 闭源通用模型，常用于长文本、代码、写作和复杂业务分析 |
| Gemini | 闭源通用模型，强调多模态、长上下文和 Google 生态集成 |
| Llama | 开权重模型，适合托管推理、私有化部署和定制 |
| Qwen | 同时提供商业服务和开权重模型，覆盖中文、代码、多模态和 Agent 场景 |
| Mistral | 同时提供商业服务和开放模型，适合托管服务与自主部署 |
| DeepSeek | 提供推理、代码和通用模型服务，也常用于成本敏感的场景 |

这张表只描述模型家族的大致定位，不能直接替代具体模型版本的测试。模型能力、价格、上下文窗口和可用功能会持续变化，实际选型应以目标模型当前文档和自己的测试结果为准。

## 模型选择维度

### 任务能力

先明确模型需要完成什么任务，例如普通问答、代码生成、文档分析、结构化抽取、复杂推理、图像理解或工具调用。

模型的通用能力排名不能代替具体任务表现。面向实际应用时，应使用自己的典型输入比较结果质量，而不是只看公开榜单或模型宣传。

### 上下文窗口

上下文窗口需要容纳 system 指令、用户输入、历史消息、检索内容和工具结果。

窗口越大不代表结果一定越好。过长上下文会增加成本和延迟，也可能让重要信息被无关内容稀释。选型时需要同时考虑最大窗口、实际可用输入长度和应用的上下文管理方式。

### Structured Outputs

如果模型输出要被程序直接解析，需要确认模型和所选 API 是否支持 JSON Schema、严格字段约束或其他结构化输出能力。

只要求模型“返回 JSON”与真正的 Structured Outputs 不是同一件事。前者仍可能出现字段缺失、类型错误或额外文本，后者能够提供更强的格式约束。

### Tool Calling

Agent 场景需要确认：

- 模型是否支持 Tool Calling
- API 是否能表达工具定义和工具调用结果
- 多工具选择是否稳定
- 参数生成是否可靠
- 是否支持并行或连续工具调用

模型支持 Tool Calling，不代表所有 Provider 和兼容 API 都完整暴露相同能力。

### 延迟

延迟至少包含首字节或首 Token 时间、完整响应时间，以及高并发时的稳定性。

交互式聊天更关注首 Token 和尾延迟，离线批处理更关注整体吞吐。只比较一次本地请求的平均耗时，无法说明真实业务中的延迟表现。

### 成本

成本不只是单次输入和输出 Token 的标价，还包括：

- Prompt 和上下文长度
- 输出长度
- 重试和 Provider 降级
- 缓存能力
- 批处理折扣
- 托管服务与自主部署的基础设施成本

低单价模型如果需要更多重试、生成更长输出或质量不足导致人工返工，最终成本不一定更低。

### 多模态与部署边界

需要处理图片、音频或视频时，应确认模型能力和 API 输入格式都支持对应模态。

涉及内网数据、数据驻留、离线运行或模型定制时，还需要考虑开权重模型和私有化部署。自主部署能够提供更多控制，但也会引入推理基础设施、容量规划、升级和运维成本。

## Provider 选择维度

模型满足任务要求后，还需要比较提供该模型的 Provider：

| 维度 | 关注内容 |
| --- | --- |
| 模型与版本 | 是否提供目标模型，模型版本是否明确，升级策略是否可控 |
| API 稳定性 | 接口兼容性、变更策略、错误结构和幂等边界 |
| 区域与网络 | 服务区域、网络延迟、跨境访问和可用性 |
| 配额与限流 | 请求速率、Token 速率、并发限制和扩容方式 |
| 计费 | 输入、输出、缓存、批处理及其他附加能力的计费方式 |
| 数据边界 | 数据保留、训练使用、隐私、合规和租户隔离策略 |
| SDK 与生态 | 官方 SDK、框架集成、文档、监控和问题排查能力 |
| 服务保障 | 可用性承诺、故障通知、技术支持和企业支持能力 |

应用需要多个 Provider 时，还要确认它们是否具备足够接近的能力。Fallback Provider 只做到“接口能调用”并不够，如果模型能力、上下文窗口或工具支持差异过大，降级后的结果可能无法满足业务要求。

## OpenAI-compatible 的边界

OpenAI-compatible 通常表示 Provider 接受与 OpenAI Chat Completions 相似的 endpoint、请求字段和响应结构，便于复用客户端和 DTO。

它不保证以下行为完全一致：

- 支持相同的请求参数
- 参数具有完全相同的取值范围和默认值
- 使用相同的模型名称
- 返回相同的错误码和错误结构
- 使用相同的限流与配额规则
- Streaming 事件完全一致
- Structured Outputs 和 Tool Calling 能力一致
- 新增的 OpenAI 专有功能可以直接使用

因此，兼容 API 可以降低基础接入成本，但每个 ProviderClient 仍需处理对应 Provider 的配置、异常和扩展差异。

## 原生 API 与兼容 API

### 适合使用兼容 API 的情况

- 主要需求是普通文本生成或聊天。
- 希望用相似的请求和响应快速接入多个 Provider。
- 需要建立统一 ProviderClient 和显式降级链。
- 当前使用的模型能力已经能通过兼容接口完整表达。
- 更看重可迁移性和较低的基础接入成本。

### 适合使用原生 API 的情况

- 需要 Provider 最新或专有能力。
- 需要完整的 Structured Outputs、Tool Calling、多模态或内置工具支持。
- 需要保留 Provider 特有的请求参数、响应元数据或错误信息。
- 兼容 API 没有暴露目标能力，或者兼容行为不稳定。
- 应用明确绑定某个 Provider，不需要为了接口统一牺牲能力。

原生 API 和兼容 API 不是全局二选一。统一 ProviderClient 可以同时包含 OpenAI-compatible、OpenAI Responses API 和 Anthropic Messages API 的实现，对上层保持统一调用边界，对下层保留各自协议能力。

## 选型顺序

模型和 Provider 可以按下面的顺序确定：

1. 明确任务、输入输出、质量要求和必须支持的能力。
2. 明确数据、部署、区域和合规等硬性约束。
3. 筛选能力和上下文窗口满足要求的模型。
4. 使用实际业务样本比较正确性、稳定性和失败情况。
5. 比较满足质量要求的模型在延迟和总成本上的表现。
6. 选择能够稳定提供目标模型的 Provider。
7. 根据所需能力和可迁移性选择原生 API 或兼容 API。
8. 如果需要降级，为候选 Provider 验证能力是否足以承接同一任务。

不存在适用于所有业务的“最佳模型”。合理的结果通常是按任务选择模型，并通过配置决定默认 Provider 和降级顺序。

## 常见场景

| 场景 | 主要关注点 | 选型方向 |
| --- | --- | --- |
| 普通问答 | 接入简单、延迟、成本 | 先选择满足质量要求的轻量模型，兼容 API 便于多 Provider 接入 |
| 复杂 Agent | 推理、Tool Calling、结构化输出 | 优先验证模型的工具调用稳定性，必要时使用原生 API 保留完整能力 |
| 长文档分析 | 上下文窗口、长文本质量、成本 | 使用真实长文档验证有效上下文和输出质量，不只比较最大窗口数字 |
| 大规模离线处理 | 吞吐、批处理、成本 | 优先比较批处理能力和单位任务总成本，交互首 Token 延迟不是主要指标 |
| 内网或敏感数据 | 数据边界、部署控制、运维 | 评估开权重模型和私有化部署，同时计算推理基础设施与运维成本 |
| 多 Provider 降级 | 契约兼容、能力接近、错误处理 | 兼容 API 降低接入成本，但必须验证降级模型能满足同一任务要求 |
