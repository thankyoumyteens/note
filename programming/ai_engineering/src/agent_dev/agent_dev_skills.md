# Agent 开发学习路线大纲

这条路线面向具备后端开发基础、希望逐步掌握 Agent 应用开发与生产落地的开发者。

学习时应围绕同一个业务案例持续迭代，而不是为每个知识点重新创建互不相关的 Demo。推荐从一个需要“理解请求、查询数据、调用工具、生成结果并由用户确认”的企业助手开始，逐阶段加入结构化输出、工具、流式响应、知识库、工作流和生产治理。

## 学习层级

- 必修主线：阶段一至阶段七。目标是能够独立设计、实现、评估和运行一个生产可用的单 Agent 应用。
- 生态扩展：阶段八。根据业务和技术栈选择，不要求掌握全部框架与平台。
- 专项加分：阶段九。面向多模态、模型微调或模型部署岗位，不是 Agent 应用开发的前置条件。

---

## 前置基础

### 开始主线前需要具备

- Java 后端基础：Spring Boot、REST API、异常处理、单元测试和集成测试
- Python 工程基础：Python、uv、基础语法和 FastAPI
- 后端工程基础：接口设计、数据库设计、日志排查和环境变量配置
- 数据基础：掌握 MySQL 或 PostgreSQL，了解 Redis
- 前端协作基础：能够看懂 TypeScript 以及 Vue 或 React 中的一种，并完成接口和流式响应联调
- 部署基础：Shell、Linux 和 Docker 的基本使用

### 在生产化阶段按需补充

- Spring Cloud、微服务、高并发、高可用、限流、熔断和降级
- MyBatis 或 JPA、Redis、Kafka 等数据访问与中间件技术
- Kubernetes、CI/CD、性能分析、监控和告警
- Flask、Django 等其他 Python Web 框架

前置基础的目标不是掌握所有后端技术，而是能够独立完成一个普通 API 服务，并具备阅读 Java 和 Python 示例的能力。

## 阶段一：最小 LLM 应用

### 学习目标

理解模型能力边界，完成稳定、可配置的模型调用，为后续能力建立统一的调用基础。

### 需要掌握

- LLM 基本原理、擅长与不擅长的任务
- Token、Tokenizer 以及 BPE、WordPiece、SentencePiece 的基本思想
- Token 数量与上下文长度、截断、推理延迟和调用成本的关系
- OpenAI-compatible API、OpenAI Responses API、Anthropic Messages API
- OpenAI、DeepSeek、Claude 等 provider 的接口差异
- model、messages、temperature、max_tokens、stream 等请求参数
- system、user、assistant 消息结构
- API Key、Base URL 和模型名的安全配置
- 统一请求、统一响应和统一 ProviderClient 抽象
- 400、401、403、429、5xx 等错误的分类处理
- 连接超时、响应超时、重试、fallback 和显式 provider 降级
- Token usage、响应延迟和调用结果的基础记录

### 实现路线

- Spring Boot + WebClient：理解底层 HTTP 协议、provider 差异和错误处理
- Spring AI：理解 Java 生态中的高层模型抽象
- Python + uv：完成轻量实验和后续评估工具准备

### 阶段产出与验收

- 同一个请求可以通过统一入口调用至少两类 provider
- 密钥与环境配置不写死在代码中，也不进入日志
- 能区分可重试错误与不可重试错误，并说明降级发生的条件
- 能记录 provider、模型、Token 用量、延迟和最终结果

---

## 阶段二：可靠的模型输出

### 学习目标

从“模型能回答”推进到“输出可以被程序稳定消费”。

### 需要掌握

#### Prompt

- Prompt 模板设计
- 角色设定、任务约束和输出格式约束
- Few-shot 示例设计
- Prompt 版本管理
- Prompt A/B 测试与持续优化

#### 结构化输出

- 用户意图拆解与任务说明
- JSON、JSON Schema、枚举和字段约束
- Schema 校验、业务校验和错误反馈
- 输出不完整、格式错误和语义错误的处理
- 输出稳定性控制

#### 幻觉与事实可靠性

- 事实型、引用型和推理型幻觉的表现与成因
- 模型不确定性、知识边界和可靠拒答
- Prompt、RAG、Tool Calling 和微调对幻觉的缓解边界
- 事实核查、引用校验、多源验证和人工复核

#### 本阶段评估

- 建立小规模、可重复运行的测试集
- 记录结构化输出成功率、字段准确率和失败样本
- 对 Prompt 或模型变更执行回归测试

### 阶段产出与验收

- 将同一业务案例的自然语言请求转换为统一结构化对象
- 非法或缺失字段不会直接进入后续业务调用
- Prompt 和 Schema 变更有版本记录
- 能用固定测试集比较变更前后的结果，而不是只靠人工试问

---

## 阶段三：最小可用 Agent

### 学习目标

让模型能够在受控边界内选择并调用工具，完成一个真实业务任务。

### 需要掌握

#### Tool Calling / Function Calling

- Tool Calling / Function Calling 原理
- Tool Schema 设计和工具描述边界
- 工具参数生成、Schema 校验和业务校验
- 外部 API 与企业内部系统集成，例如 Jira、飞书、KMS、PLM 和业务系统 API
- 工具调用结果解析、摘要和重新注入上下文
- 工具调用失败的分类、重试和终止
- 防止模型调用不存在、不适用或越权的工具

#### Agent 基础循环

- Agent 基本架构
- Direct、ReAct、Plan-and-Execute 和 Reflection 的基本差异与适用边界
- 根据任务复杂度、工具依赖、可靠性和成本选择执行模式
- ReAct 模式和受限的“判断—调用—观察—继续”循环
- 根据工具执行结果调整剩余计划，避免无效反思和重复执行
- Tool、Context、State 和 Runtime 的基础职责
- Agent step 限制和终止条件
- Human-in-the-loop 人工确认机制

#### 工具管理与选择

- 工具注册、分类、版本和生命周期管理
- 工具较多时的分层加载、动态发现和候选工具筛选
- 基于规则、意图或语义检索选择候选工具
- 只将候选工具 Schema 注入上下文，控制 Token 消耗和误调用
- 工具路由准确率、Top-K 召回率和误调用率评估

#### 本阶段治理

- 工具白名单、最小权限和高风险操作确认
- 工具输入、结果、耗时、异常和调用者记录
- 工具调用成功率、参数正确率和任务完成率评估

### 阶段产出与验收

- Agent 可以根据请求选择正确工具并完成至少一个端到端业务任务
- 所有工具参数在执行前经过校验
- 只读、可逆和高风险操作具有明确的权限与确认边界
- Agent 不会无限循环，失败后能返回可解释的终止结果

---

## 阶段四：Streaming / SSE 与统一响应协议

### 学习目标

在已有业务能力上增加稳定的流式交互，而不是把流式传输与 Agent 核心逻辑混在一起。

### 需要掌握

- Streaming 与 SSE 的基本概念及区别
- OpenAI-compatible、Responses API 和 Anthropic Messages API 的流式事件解析
- 使用 Java 和 Python 实现流式输出
- 前端使用 fetch + ReadableStream 接收 POST SSE
- 后端统一封装前端事件协议
- message、tool、error、done 等事件设计
- 首 Token 超时、流中空闲超时和总时长超时
- 代理缓冲、心跳、客户端断流、资源释放和错误事件处理
- 重连、恢复与断点续传的适用边界
- 流开始前与开始后的重试和 provider 降级边界

### 阶段产出与验收

- 同一业务能力同时支持非流式和流式响应
- 前端不需要理解不同 provider 的原始事件格式
- 客户端断开后，后端能停止上游调用并释放资源
- 流中错误通过统一事件表达，不会伪装成正常完成

---

## 阶段五：知识与 Context Engineering

### 学习目标

让 Agent 能够使用外部知识，并在有限的上下文窗口内选择可靠、相关的信息。

### 需要掌握

#### RAG 知识库系统

- PDF、Word、Markdown、HTML、网页和表格解析
- Chunking、overlap 和元数据设计
- Embedding 模型选型与向量化流程
- 余弦相似度、点积、欧氏距离和近似最近邻检索的适用边界
- pgvector、Milvus、Chroma、FAISS、Pinecone 等向量存储的选型边界
- 关键词检索、Elasticsearch 和混合检索
- Top-K、相似度阈值、MMR、Query Rewrite、召回优化和 Rerank
- Embedding 批处理、缓存、增量更新和模型版本变更后的缓存失效
- 检索缓存与生成结果缓存的适用边界
- 引用、来源定位和上下文拼接
- 知识库更新、删除、权限、租户隔离和版本管理

#### Context Engineering

- Token 预算、上下文窗口、成本与延迟控制
- System Prompt、历史对话、RAG、工具结果和状态的优先级
- 上下文选择、排序、裁剪、摘要和截断
- 对话状态、短期记忆与长期记忆
- 用户画像与个性化上下文
- 上下文可信度标注、冲突处理和可靠回答边界
- Prompt Injection 与检索内容污染防护

#### Memory 工程

- 记忆写入、读取、更新、合并、过期和删除的生命周期
- 基于重要性、相关性、时效性和重复度的记忆写入策略
- 用户明确事实与模型推断信息的分离存储和可信度标注
- 记忆冲突检测、版本替换、历史保留和可撤销更新
- 原始对话、摘要、结构化记忆和向量记忆的适用边界
- 多会话、多任务和多 Agent 下的命名空间与权限隔离
- 并发更新、乱序写入、重复写入和一致性处理

#### 本阶段评估

- 为检索建立独立评估集，区分检索质量和最终回答质量
- 评估 Recall、Precision、Rerank 效果、引用正确性和无答案处理
- 记录失败查询并分析解析、切分、召回、排序或生成环节的问题

### 阶段产出与验收

- 回答可以追溯到具体知识来源
- 不同用户只能检索其有权访问的数据
- 找不到可靠证据时能够明确拒答或说明信息不足
- 能用评估集比较切分、检索和 Rerank 策略，而不是只凭主观感受

---

## 阶段六：Workflow、状态机与 Agent 架构

### 学习目标

将单次工具调用扩展为可追踪、可恢复、可人工介入的多步骤任务。

### 需要掌握

#### Workflow / 状态机 / LangGraph 思想

- 多步骤任务流设计
- 固定工作流与动态 Agent 的区别和选型
- 状态机、节点、边、状态和条件分支
- 任务重试、跳转、暂停、恢复和终止
- 工作流异常处理和模板复用
- 状态持久化、幂等、检查点和失败恢复
- 执行轨迹记录

#### Agent 架构设计

- Planner、Executor、Tool、Memory、Context 和 Runtime 的职责边界
- 任务规划、任务拆解和执行链路
- Agent 状态管理
- Human-in-the-loop 的暂停、审批和恢复
- Tool、Memory、Context 和 Runtime 抽象
- Agent 失败恢复与容错机制

#### 路由设计

- 意图路由、工具路由、Agent 路由和 Workflow 分支路由的职责边界
- 规则、分类模型、LLM 和向量检索路由的适用场景
- 路由低置信度时的澄清、拒绝和回退机制
- 使用准确率、Top-K 召回率、混淆矩阵和用户纠正率评估路由效果

### 阶段产出与验收

- 用状态图表达一个包含分支、重试和人工确认的业务流程
- 服务重启或步骤失败后，可以从安全检查点恢复
- 重试不会重复执行不可重复的外部操作
- 能解释为什么某个场景使用固定工作流、动态 Agent 或两者组合

---

## 阶段七：Eval、Trace、安全与生产化

### 学习目标

把前面各阶段的局部治理能力整合为可持续评估、观测、审计和运营的生产系统。

### 需要掌握

#### Eval

- 准确率、任务完成率和响应延迟评估
- 工具调用成功率和 RAG 检索指标
- 自动化测试集构建、数据版本和标注规范
- 回归测试、失败案例分析和发布门禁
- LLM-as-Judge 的适用范围、偏差和人工校准

#### Trace / Observability / LLMOps

- Agent 执行链路 Trace
- Prompt log、tool trace 和 conversation replay
- Token usage、成本、延迟、错误率和稳定性指标
- Prompt、模型、工具、知识库与评估集版本关联
- 监控告警、问题定位和审计记录

#### 安全、权限、成本与稳定性

- Tool 权限、数据访问权限和企业内部系统安全接入
- Prompt Injection、RAG 数据污染和不可信工具结果防护
- 敏感信息过滤、日志脱敏和 API Key 保护
- 限流、熔断、降级、超时和重试
- 高并发下的资源隔离与稳定性保障
- Token 配额、成本限额、Agent step 和执行时长限制
- 审批、审计和应急终止机制

#### Guardrails

- 输入护栏：Prompt Injection、文件类型、长度和敏感内容检查
- 执行护栏：权限、参数、调用次数、Token 预算和沙箱限制
- 输出护栏：敏感信息、事实依据、格式和业务规则检查
- 护栏误杀率、漏检率和人工复核边界

#### 部署与运维

- Docker、CI/CD、环境隔离和配置管理
- 根据部署规模按需使用 Redis、Kafka、Kubernetes 和分布式治理能力

### 阶段产出与验收

- 每次回答能够关联 Prompt、模型、工具、知识来源、Trace 和评估版本
- 核心质量指标下降时能够通过回归测试或监控发现
- 成本、并发、执行步数和高风险操作都有明确上限
- 能从失败结果定位到模型、检索、工具、工作流或基础设施环节

## 阶段八：框架、平台与协议生态

### 学习目标

在掌握框架无关的核心能力后，根据业务需要选择生态工具，避免为了使用框架而增加系统复杂度。

### 主线实现选择

- Spring Boot + WebClient：适合需要完全控制协议、provider 和治理逻辑的 Java 项目
- Spring AI：适合以 Spring 生态为主的 Agent 应用
- Python + uv：适合实验、评估、RAG 和 LangGraph 等 Python 生态
- Dify：适合低代码原型验证和标准化业务流程

不要求在每个生产项目中同时使用以上所有实现；学习文档中的代码知识点仍分别覆盖三种实现方式。

### 按场景学习

- LangChain：Python LLM 应用组件与生态
- LangGraph：有状态 Agent 工作流
- LlamaIndex：RAG、数据连接和检索增强
- MCP Server、MCP Client、Tools 暴露和标准化连接器
- Skills 机制和企业内部能力封装
- Dify、Coze、Flowise、n8n、FastGPT 等低代码或工作流平台

### 了解与扩展

- A2A / Agent-to-Agent 基本思想和安全接入机制
- AutoGen、AgentScope、MetaGPT 等多 Agent 框架
- AutoGPT、BabyAGI、Hermes-Agent 等开源项目的基本思路

### 多 Agent 协作

只有在单 Agent 无法通过工具拆分或固定工作流解决问题时，再学习和使用多 Agent：

- Coordinator / Worker、Supervisor 和专家 Agent 模式
- 任务分派、信息传递和协作协议
- 冲突处理和共享状态同步
- 多 Agent 的权限边界、失败隔离、成本和延迟控制

### 阶段产出与验收

- 能基于具体约束说明框架或平台的选型理由
- 能通过 MCP 或其他明确协议暴露和接入一个受控工具
- 能判断一个问题是否真的需要多 Agent
- 引入框架后仍保留业务状态、权限、Trace 和评估边界

## 阶段九：模型训练、微调与多模态专项

本阶段属于加分项，不是 Agent 应用开发的必修前置条件。

### 需要掌握

- 基础机器学习和深度学习
- PyTorch、sklearn
- 训练数据、验证集和测试集构建
- 训练数据清洗、去重、脱敏、版本管理和质量评估
- SFT、偏好对齐、LoRA、QLoRA 及其适用边界
- 灾难性遗忘、跨领域泛化、离线评估、线上评估和回滚
- 模型量化、蒸馏、裁剪和部署
- 推理吞吐量、首 Token 延迟、生成速度和并发能力
- 动态批处理、Continuous Batching 和 KV Cache
- 模型参数量、上下文长度、GPU 显存和硬件成本的关系
- 云端 API、私有化部署和边缘部署的选型边界
- 垂直领域模型优化
- 多模态模型基础
- 文本、图像、音频、视频与 Agent 的组合
- CV + Agent 场景

### 阶段产出与验收

- 能判断问题应通过 Prompt、RAG、Tool Calling 还是微调解决
- 微调前有明确基线、数据集和评估指标
- 多模态输入有明确的数据权限、质量和失败处理边界

## 贯穿路线的产品落地要求

产品落地不是最后单独学习的章节，而是每个阶段都要推进的主线。

需要持续练习：

- 从业务需求拆解 Agent 目标、工具、数据、流程和风险
- 在智能客服、企业知识库、办公自动化、金融、汽车研发、健康管理或企业流程中选择一个主要案例
- 与前端、后端、算法和产品团队明确接口及责任边界
- 输出技术方案、评估报告、运行手册和安全说明
- 沉淀可复用的 ProviderClient、Tool、Context、Workflow、Eval 和 Trace 组件

最终验收不是“使用了多少框架”，而是能否交付一个有明确业务价值、质量可评估、执行可追踪、权限可控制、失败可恢复、成本可约束的 Agent 应用。
