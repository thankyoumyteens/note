# Agent Workflow是什么

**Agent** 可以先理解成：

```text
能根据目标，自己决定下一步该做什么，并调用工具完成任务的 AI 程序。
```

它不是单纯聊天，而是会执行一个任务流程。

比如普通 LLM 是：

```text
用户问问题 -> 模型回答
```

Agent 是：

```text
用户给目标
  -> 模型分析目标
  -> 决定下一步
  -> 调用工具 / 查数据库 / 查订单 / 检索知识库
  -> 根据结果继续判断
  -> 直到完成任务或需要人工介入
```

举个例子，用户说：

```text
帮我处理这个客户退款工单。
```

普通模型可能只会写一段建议。

Agent 会做：

```text
1. 分析工单
2. 判断需要查订单
3. 调用订单查询工具
4. 发现退款状态异常
5. 生成客服回复草稿
6. 如果有风险，交给人工审核
```

---

**Agent Workflow** 是 Agent 的工程化版本。

它不是让 Agent 无限自由发挥，而是把 Agent 限制在一个明确流程里：

```text
创建工单
  -> 生成计划
  -> 检查订单
  -> 生成回复草稿
  -> 等待人工审核
  -> 完成 / 失败
```

也就是你当前第 16 课里的核心设计：

```text
状态机 + 有限步骤 + 可恢复 + 可人工介入
```

这比“自由循环 Agent”更适合企业系统，因为它更可控。

---

两者关系可以这样理解：

```text
Agent = 会思考和调用工具的 AI 执行者

Agent Workflow = 给 Agent 套上明确流程、状态和边界的工程系统
```

更直白一点：

```text
Agent 像一个 AI 助手。
Agent Workflow 像给这个 AI 助手制定的 SOP 流程。
```

为什么课程不直接做“自由 Agent”？因为风险很高：

```text
可能无限循环
可能乱调用工具
可能成本失控
可能越权操作
失败后不好恢复
中间过程不好审计
```

所以第 16 课先做更安全的 Workflow Agent：

```text
每次只推进一步
有 maxSteps 限制
有明确 status
有明确 currentStep
需要人工确认时暂停
失败进入 FAILED
```

这就是企业级 Agent 的正确入门方式。

## 状态机

**状态机**就是：

```text
一个系统在任何时刻都处于某个明确状态，并且只能按照规定规则从一个状态切换到另一个状态。
```

它有三个核心要素：

```text
状态 State
事件 Event
状态转移 Transition
```

比如一个订单系统：

```text
CREATED
  -> PAID
  -> SHIPPED
  -> COMPLETED
```

如果用户付款，就从：

```text
CREATED -> PAID
```

如果发货，就从：

```text
PAID -> SHIPPED
```

但不能直接：

```text
CREATED -> COMPLETED
```

因为这个转移不符合规则。

---

## 为什么 Agent 需要状态机？

如果没有状态机，Agent 很容易变成：

```text
模型自己决定下一步
模型自己决定是否继续
模型自己决定调用什么工具
模型自己决定什么时候结束
```

这在企业系统里风险很大：

```text
可能无限循环
可能重复调用工具
可能跳过人工审核
可能失败后继续乱跑
不好恢复
不好审计
```

状态机的作用就是给 Agent 加边界：

```text
当前在哪一步
下一步只能去哪
失败怎么办
什么时候暂停
什么时候结束
最多能走几步
```

一句话总结：

```text
状态机 = 用明确状态和转移规则控制流程的机制。
```

在 Agent Workflow 里，它的作用是：**把不稳定的 AI 决策限制在可控、可审计、可恢复的业务流程中。**
