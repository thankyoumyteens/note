# MCP 协议与企业级架构融合

作为 Java 后端开发者，如果要用一句话向你解释什么是 MCP，那就是：MCP 就是大模型世界的 JDBC。

在没有 JDBC 的黑暗时代，你的 Java 程序如果要连 MySQL，就要写一套专有的 MySQL 驱动代码；要连 Oracle，又要重写一套 Oracle 的底层通信代码。这叫 N × M 的灾难。

在目前的 AI 时代，大模型连接外部数据也面临一模一样的灾难：如果你写了一个 Agent，你想让它读取本地电脑的文件、搜索公司的 Confluence 文档、查询内部的 Spring Boot 订单系统，你必须为每一个数据源手写一个 `@tool`。如果明天公司把 Qwen 换成了 Claude，你可能又要把底层的 Tool Calling 解析逻辑重写一遍。

MCP (Model Context Protocol) 诞生，就是为了制定标准协议，实现 1 × N 的一劳永逸。

## MCP 的三层核心架构

MCP 采用了极其经典的 Client-Server (C/S) 架构。它强制把“大脑”和“手脚”在物理和网络边界上切开了。

| 架构层级                | 现实对应                                                 | Java 后端视角对标                          |
| :---------------------- | :------------------------------------------------------- | :----------------------------------------- |
| **MCP Host (宿主)**     | 你刚写的 LangGraph 脚本、或者 Claude Desktop 桌面端软件  | **Spring Boot 业务主容器**                 |
| **MCP Client (客户端)** | 宿主内部用来发起协议请求的 SDK 代码                      | **JDBC 接口 / Mybatis Mapper**             |
| **MCP Server (服务端)** | 运行在你公司内网、贴着旧数据库和遗留系统部署的轻量级服务 | **数据库驱动 (MySQL Driver) / 底层微服务** |

## 为什么这对“遗留系统”是巨大的利好？

假设你们公司有一个祖传的 Java 订单系统（不能乱改代码，只对外暴露了几个古老的 REST API）。

过去的做法：你得在 Python 的 LangGraph 代码里，写一堆 `requests.get()` 去调那些古老的 API，还要处理各种权限和数据清洗，Python 代码瞬间变得极其臃肿。

MCP 的做法：你在内网用任何语言（甚至就用 Java 自己）写一个极其轻量级的 MCP Server。这个 Server 把祖传 API 包装成了 MCP 协议能懂的标准格式。你的 LangGraph Agent 只需要通过网络连上这个 MCP Server，大模型就能瞬间“看懂”你们公司的订单数据。

## MCP Server 暴露的“三大神兵”

当一个 MCP Client 连上 MCP Server 时，Server 会向大模型暴露三种核心能力（这三点是 MCP 协议的灵魂）：

- Resources (资源)：读操作。类似大模型的“只读网盘”。你可以把公司内部的 API 接口、本地 log 文件、数据库表包装成 Resource。大模型可以随时通过 URI 去读取它们作为上下文。
- Tools (工具)：写操作/执行操作。这就相当于我们手写的 `@tool`。它可以是触发一个转账动作，或者在遗留系统中新建一条记录。
- Prompts (提示词模板)：服务端可以把自己沉淀好的优质 Prompt 模板暴露给客户端。这意味着逻辑被后置了，前端的 Agent 不需要知道复杂的业务怎么写，直接向 Server 请求对应的模板即可。

MCP 让你彻底告别了在 Agent 代码里写死业务逻辑的时代。你的 LangGraph 现在只需要专注一件事：思考和流转。所有的脏活、累活、查数据库的活，全部通过 MCP 协议外包给后端的各种 MCP Servers。
