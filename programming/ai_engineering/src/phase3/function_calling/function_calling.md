# 掌握 Function Calling 与单体 Agent 原理

在纯聊天模式下，大模型就像一个“被绑在椅子上的全知智者”，能回答问题，但无法改变现实。Function Calling 就是你递给它的那把瑞士军刀。

## Function Calling 的三次握手（API 契约）

很多初学者误以为是“大模型自己去调用了 API”。绝对不是！ 大模型由于处于沙盒中，它没有任何发起网络请求的能力。真实的运行机制是一个极其优雅的“抛接球”过程：

1. 抛出契约 (System -> LLM)： 你在请求大模型时，除了传入用户的对话，还在 `tools` 参数里传入了一堆你本地写好的函数的“说明书”（JSON Schema，包含函数名、描述、入参类型）。
2. 大模型下发指令 (LLM -> System)： 大模型发现用户的提问需要用到某个工具（比如查天气）。它会停止输出自然语言，而是返回一段极其标准的 JSON 给你的后端代码：
   ```json
   "tool_calls": [
        {
            "name": "get_weather",
            "arguments": "{\"city\": \"北京\"}"
        }
    ]
   ```
3. 本地执行与结果回传 (System -> LLM)： 你的 Java/Python 后端代码拦截到这个 JSON，利用反射或路由分发，执行本地真正的 `get_weather("北京")` 函数。拿到结果（如“晴，25度”）后，将这个结果再次拼接到对话历史中，重新请求大模型，大模型最后根据这个结果生成自然语言回答用户。

## ReAct 框架：让模型拥有“内心戏”

ReAct（Reason + Act）是单体 Agent 最经典的运行逻辑。它要求模型在采取行动前，必须先“自言自语”输出思考过程。这相当于强制让大模型打印它的 DEBUG 日志。

标准闭环如下：

- Thought (思考)： “用户问北京今天需要带伞吗。我需要先获取北京的天气。”
- Action (行动)： 输出调用 `get_weather` 的 JSON 指令。
- Observation (观察)： 接收后端返回的执行结果：“晴天”。
- Thought (思考 2)： “天气是晴天，所以不需要带伞。”
- Final Answer (最终回答)： “北京今天是晴天，不用带伞哦。”
