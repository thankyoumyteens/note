# Langfuse

Langfuse 是专为大语言模型（LLM）打造的开源可观测性平台。在 AI 时代，大模型的输入输出充满不确定性。Langfuse 就是 AI 系统的“黑匣子”和“塔台雷达”，让“玄学”调优变成数据驱动的工程科学。

核心 Feature:

1.  **全链路追踪 (Tracing)**：记录 AI 响应的 **Latency (延迟)**。通过 Span 和 Generation 节点，一眼定位是 Java 接口背锅还是模型幻觉。
2.  **提示词解耦 (Prompt Management)**：通过 SDK 动态拉取后台管理的 Prompt，实现“代码不动，Prompt 升级”。
3.  **量化闭环 (Evaluation)**：将 Token 成本、TTFT（首字延迟）和用户评分进行可视化大盘展示，从“玄学”走向“量化”。

## 1. 世代演进：从 V2 到 V3 的跨越

### V2 时代：小作坊的轻骑兵

- **背景**：早期为了快速普及，采用 **PostgreSQL** 存储一切。
- **特点**：**同步写入 (Synchronous)**。SDK 发送数据，Server 必须实时写入磁盘后才返回响应。
- **局限**：当并发流量升高（TPS > 100），Postgres 的磁盘 IO 会被打爆，导致业务系统（Python 脚本）因为等待 Trace 上传而变慢。

### V3 时代：工业级的航空母舰

- **背景**：为了应对亿级流量，官方引入了 **OLAP（列式存储）** 大数据架构。
- **特点**：**异步解耦 (Asynchronous)**。Server 收到数据先扔进 MinIO（对象存储）和 Redis（队列），由后台 Worker 慢慢搬运到 ClickHouse。
- **核心组件 (七神编队)**：
  1.  **PostgreSQL**: 只负责元数据（用户、配置、Prompt 模板）。
  2.  **ClickHouse**: 负责海量 Trace 数据存储与疯狂的聚合分析。
  3.  **ZooKeeper**: 集群协调员（Schedule 数据同步）。
  4.  **MinIO**: 异步暂存区（S3 存储桶）。
  5.  **Redis**: 消息总线（通知 Worker 搬砖）。
  6.  **Server**: 前台指挥官。
  7.  **Worker**: 异步搬砖工（核心性能引擎）。

## 2. V2 vs V3 深度对照表

| 维度         | V2 (Postgres 核心)     | V3 (分布式集群)                     |
| :----------- | :--------------------- | :---------------------------------- |
| **底层架构** | 单体 OLTP              | 分布式 OLAP (ClickHouse)            |
| **写入机制** | 同步 (直连磁盘)        | 异步 (缓冲区削峰填谷)               |
| **查询性能** | 面对海量文本查询较慢   | 亿级数据量秒级聚合                  |
| **硬件要求** | 极低 (1-2GB 内存)      | 极高 (建议 12GB+ 内存) |
| **使用场景** | **原型开发、本地调试** | **生产环境、C端高并发应用**         |
